# SpireX/DI — v1.2.0 Change Log

## Internal Module Types
v1.2.0 introduces the ability to define an Internal `TypeMap` for modules. Types declared in this map are only visible within the module itself and are not accessible from other modules or the container where the module is included. This allows developers to hide services that are necessary for the operation of a module’s public services, improving encapsulation and modularity.

```ts
const MyModule = staticModule("MyModule").create<{
    // PUBLIC
    myService: MyService
}, {
    // INTERNAL
    myServiceGateway: IMyServiceGateway
    myServiceRepo: IMyServiceRepository
}>((binder) => {
  // TypeMap: { myService, myServiceGateway, myServiceRepo }
})

const builder = diBuilder()
    .include(MyModule)
// TypeMap: { myService }
```

Additionally, the `internal` option ensures these types are **completely inaccessible** at runtime, preventing external access even if TypeScript checks are bypassed.

```ts
const MyModule = staticModule("MyModule").create(binder => {
    binder
        .bindInstance("mySecret", "JdQtcdZGmeHs", { internal: true })
        .bindFactory("myRepo", r => new MyRepository(r.get("mySecret")), { internal: true })
        .bindFactory("myService", r => new MyService(r.get("myRepo")))
})
```

## Module Composition

You can now create **Composite Modules** by grouping multiple modules into a single module. Composite modules aggregate the types of all included modules while keeping each module’s individuality intact. This makes it easier to configure containers and reuse commonly combined modules without losing access to their type information.

```ts
const MarketingModule = staticModule("Marketing").compose(
    SplashesModule,
    PaywallsModule,
    OffersModule,
)

builder.include(MarketingModule)
```

Composite modules can also be nested in higher-level compositions, enabling modular, hierarchical structures:

```ts
const MonetizationModule = staticModule("Monetization").compose(
    MarketingModule,
    InAppPurchasesModule,
    AccessModule,
)
```


## Resolution Stack in Hooks
Middleware hooks `onRequest` and `onResolve` now receive the type **resolution stack**. This stack allows developers to inspect the chain of type requests leading to the current resolution. If the stack is empty, the instance is requested directly from the container or scope.

```ts
const builder = diBuilder().use({
    onRequest(entry, scope, type, name, resolutionStack) {},
    onResolve(entry, instance, scope, resolutionStack) { return instance },
})
```

## Local Scope Data
Scopes can now hold **local, immutable data** at creation. This allows developers to store request-specific, session-specific, or context-specific information safely.

```ts
const builder = diBuilder({ data: { debug: true } })

const requestScope = container.scope("request", { data: { uri: "/path/to/method" } })
```

Local data is accessible from factories, middleware, or directly from the scope or container:
```ts
builder.bindFactory(
  "handler",
  (r, ctx) => new RequestHandler(ctx.data.uri),
  { lifecycle: "scope" },
)

console.log("DEBUG: ", container.data?.debug) // true
console.log("Request URI:", requestScope.data?.uri) // "/path/to/method"
```


## Alias Binding Hooks
A new hook `onBindAlias` allows middleware to intercept alias bindings. This provides visibility into all bindings, including those created via aliases, enabling better monitoring and customization of dependency resolution.

```ts
builder.use({
    onBindAlias(alias, origin, builder) { ... },
})
```

## Alias Search in Builder
A new `findAlias` method allows developers to search for aliases in the builder using a predicate. This supports controlled traversal with continue/break behavior.

```ts
var alias = builder.findAlias(({ type, name }) => type.endsWith("Handler") && name)
```

## Package Size Optimization
Despite adding multiple new features, the library size has been optimized to maintain its lightweight status. The current size is **8.7Kb uncompressed (3.58Kb gzip)**, down from 10.8Kb before optimization, without affecting container interfaces or performance.

For comparison with previous releases:
- v1.0.0 — 9.18Kb
- v1.1.0 / v1.1.1 — 9.76Kb


------------------
## Summary

v1.2.0 significantly enhances modularity, encapsulation, and observability in SpireX/DI. Internal module types can now be fully hidden both at the TypeScript and runtime level, while module composition simplifies container configuration and aggregates types. Middleware gains insight into type resolution chains, and scopes can store immutable, context-specific data. Alias hooks and search functions provide better control over binding behavior. Finally, the library remains compact and efficient despite these new capabilities.
