# SpireX/DI — v1.2.2 Change Log

## Scope data inheritance
Fixed a bug where local data defined on a container or parent scope was not inherited by child scopes. Previously, creating a child scope from a parent that had local data resulted in the child having no data.

Example before:
```ts
const container = diBuilder().build();

const scopeA = container.scope('A', { data: { value: 'foo' } });
// scopeA.data = { value: "foo" }

const scopeB = scopeA.scope('B');
// scopeB.data = undefined
```

Now a child scope inherits the parent's local data when no local data is supplied to the child:
```ts
const scopeA = container.scope('A', { data: { value: 42 } });
const scopeB = container.scope('B');

// scopeB.data = { value: 42 }
```

When a child scope is created with its own local data, the runtime merges the parent and child local data into a new object so both sets of values are available:
```ts
const scopeA = container.scope('A', { data: { value: 42 } });
const scopeB = container.scope('B', { data: { msg: 'Hello' } });

// scopeB.data = { value: 42, msg: 'Hello' }
```

If the child scope is created with isolation enabled, it does not inherit parent data; only the child’s local data is used:
```ts
const scopeA = container.scope('A', { data: { value: 42 } });
const scopeB = container.scope('B', { data: { msg: 'Hello' }, isolated: true });

// scopeB.data = { msg: 'Hello' }
```

The base type for scope data is now declared as object to ensure safe merging behavior when inheriting and combining parent and child data.


## Named binding scope identifier

Fixed scope identifier generation for named bindings created with `withScope`. Previously the identifier used only the type tag and ignored the binding name, which prevented bindings of the same type but different names from owning distinct scopes. Now the binding name is included in the identifier, allowing unique scopes per named binding.

```ts
builder.bindFactory('service', factoryOf(ExperimentalService), { name: 'exp', withScope: true });
```


## Middleware hook invocation optimization
Optimized middleware hook invocation so that only middleware instances which implement the specific hook are iterated and called. The runtime no longer traverses all middleware for every hook invocation, reducing unnecessary checks and improving performance.


## Factory activation context

Factory activation now receives expanded context information. The activation context exposes a readonly `stack` containing the current activation stack (so a factory can inspect what triggered its activation) and `name` containing the current binding name. These values are available on the second argument's ctx property.

```js
var container = diBuilder()
    .bindFactory("service", (r, { ctx }) => {
        // ctx.name = 'foo'
        // ctx.stack = [{ type: 'manager', ... }]
        return new Service(ctx.name)
    }, { name: 'foo' })
    .bindFactory('manager', r => new Manager(r.get('service', 'foo')))
    .build()

var manager = container.get('manager');
```
