# SpireX/DI — v1.1.1 Change Log

## Fixed: Correct Handling of Falsy Instance Bindings

Previous versions of SpireX/DI incorrectly handled falsy values when binding instances.
If a falsy value was provided, the container treated it as a missing factory and attempted to invoke it, resulting in a runtime error.

```ts
builder.bindInstance('featureFlag', false);

container.get('featureFlag');
// Error: 'factory' is not a function
```

All falsy values except `undefined` are now supported and will be stored and returned as-is:
`null`, `false`, `""` (empty string), `0`, `-0`, `0n`, `NaN`.

```ts
builder.bindInstance('retryCount', 0);
builder.bindInstance('isEnabled', false);

container.get('retryCount'); // 0
container.get('isEnabled');  // false
```

Binding `undefined` as an instance value is still prohibited. Attempting to do so will throw a runtime error:
```ts
builder.bindInstance('service', undefined);
// TypeError: Cannot bind undefined to type "service".
```

This restriction applies **only to instance bindings**.
Factories are still allowed to return `undefined` without any issues, which is useful for optional or lazy-resolved dependencies.

## Fixed: Middleware TypeMap Propagation
Version 1.1.0 introduced typed middleware, enabling middleware to extend the container’s `TypeMap`.
However, there was a flaw in the internal typing: middleware hooks (`onUse`, etc.) received the **original** `TypeMap`, not the extended one.

As a result, attempting to bind or resolve newly introduced types inside middleware
caused TypeScript errors, even though the behavior was valid at runtime.

This issue has been resolved. The updated `TypeMap` is now correctly propagated to **all middleware hooks**.

```ts
const MyMiddleware = <TypeMap>(): IContainerMiddleware<
    TypeMap,
    MiddlewareTypeMap
> => ({
    onUse(builder) {
        // before: IContainerBuilder<TypeMap>
        // after:  IContainerBuilder<TypeMap & MiddlewareTypeMap>

        builder.bind('newService', () => ({ /* ... */ }));
    }
});
```

This fix ensures that TypeScript accurately reflects the container state throughout the entire middleware lifecycle.


## Fixed: Module and Middleware Deduplication by Name
Previously, the container builder determined whether a module or middleware was already included by comparing references only.
This caused problems when modules or middleware were created via factory functions, as each invocation produced a new instance.

Even if these instances represented the same logical module,
the builder treated them as distinct and included them multiple times.

The inclusion check has been improved:
- **Modules and middleware are now deduplicated by name**
- If two modules share the same name, they are treated as the same module
- Middleware names remain optional

For unnamed middleware, the builder falls back to reference comparison to preserve backward compatibility.

This change prevents accidental duplicate registrations
while still allowing flexible middleware composition through factories.

--------------
## Summary

Version 1.1.1 focuses on correctness and developer experience:
- Falsy instance bindings now behave predictably at runtime
- Middleware type extensions are fully reflected in TypeScript
- Container composition is more robust when using factories

No breaking changes were introduced, but several subtle edge cases have been resolved
to make dependency management safer and more intuitive.
