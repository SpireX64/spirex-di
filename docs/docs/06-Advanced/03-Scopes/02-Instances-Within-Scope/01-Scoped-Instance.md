# Экземпляры ограниченные областью

Теперь зная как создавать области, можно разобраться о том как ведут себя экземпляры типов при взаимодействии с ними.

Как было сказано, области имеют полный перечень операций для получения экземпляров, аналогичный самому контейнеру: `get`, `maybe`, `providerOf` и т.д. Что дает возможность запрашивать экземпляры непосредственно из самой области.

Однако в зависимости от жизненного цикла экземпляра, логика получения экземпляра может изменяться.


Для начала рассмотрим самый простой случай привязки - привязка экземпляра. Этот вид привязки всегда возвращает одно и тоже значение и известен еще до создания самого контейнера. Если запросить экземпляр такого типа из области, то он просто вернет это привязанное значение. Область ни как не повлияет на это.

```ts
builder.bindInstance("value", "foo")

container.scope("domain").get("value") // "foo"
```

Экземпляры с жизненным циклом "transient" также не управляются контейнером и имеют такое же поведение как и у простой привязки экземпляра. При запросе такого экземпляра контейнер создает его и сразу отдает без какого-либо кеширования на уровне контейнера или области.

```ts
builder.bindFactory(
    "username", 
    r => r.get("UserService").currentUser?.name,
    { lifecycle: "transient" },
)

container.get("username")
container.scope("foo").get("username")
```


Теперь рассмотрим привязку фабрик с жизненными циклами "singleton" и "lazy". Оба этих жизненных цикла утверждают, что в контейнере будет существовать только один экземпляр типа. Экземпляры типов с этими жизненными циклами хранятся в коренной области - контейнере. Что гарантирует соблюдение контракт единственного экземпляра.

При попытке получить экземпляр с жизненным циклом "singleton" или "lazy" с помощью области, будет выполнено обращение напрямую к коренной области минуя всю иерархию областей. Что дает очень быстрое получение таких экземпляров, так как это не требует выполнять доп. проверки для разрешения экземпляра.

```ts
builder.bindFactory(
    "UserService",
    factoryOf(UserService),
    { lifecycle: "lazy" },
)

var serviceA = container.get("UserService")
var serviceB = container.scope("foo").scope("bar").get("UserService")

assert(serviceA === serviceB)
```


И наконец жизненный цикл "scope", который как раз и существует для управления экземплярами с помощью областей. Этот жизненный цикл работает аналогично "lazy", за исключением того, что созданный экземпляр будет хранится и подчиняться области в которой он создан.

При запросе экземпляра с жизненным циклом "scope" из области происходит следующее:
1. Проверяется, нет ли уже существующего экземпляра в текущей области.
2. Если нет, поиск ведется вверх по всей иерархии областей до самого корня.
3. Если экземпляр так и не был найден, выполняется его создание и далее сохранение в области из которой он был запрошен.

Однако это поведение может быть немного изменено с помощью введения некоторых правил, которые будут рассмотрены в отдельных разделах.

```ts
builder.bindFactory(
    "PageEditorStore",
    factoryOf(PageEditorStore),
    { lifecycle: "scope" },
)

const scopePageEditor = container.scope("page-editor")
const store = scopePageEditor.get("PageEditorStore")

const scopePageEditor2 = container.scope('page-editor-2')
const store2 = scopePageEditor2.get("PageEditorStore")

// Разные экземпляры в областях разной иерархии
assert(store !== store2)
```

Внимание, так как контейнер является коренной областью, после получения экземпляра с жизненным циклом "scope" на уровне контейнера, он получает жизненный цикл аналогичный "lazy". То есть будет существовать на протяжении всей жизни самого контейнера. Однако все еще будет иметь дополнительные проверки и обходы, которые специфичны для "scope" экземпляров, что не бесплатны и отсутствуют в экземплярах с жизненным циклом "lazy".

После того как экземпляр был создан в области, он ведет себя аналогично "singleton" и "lazy" в пределах этой области. То есть всегда будет возвращаться один и тот же экземпляр.

Возможность получить экземпляры из родительских областей дает возможность иметь только один общий экземпляр для текущей иерархии. Так как такой экземпляр скорее всего уже инициализирован и имеет все необходимые данные. Создавать по такому сервису на каждую область было бы очень неудобно и расточительно.

```ts
const scopePageEditorModal = scopePageEditor.scope("modal")
const storeAtModal = scopePageEditorModal.get("PageEditorStore")

const scopePageEditorActions = scopePageEditorMonitor("actions")
const storeAtActions = scopePageEditorActions.get("PageEditorStore")

// Один и тот же экземпляр в дочерних областях
assert(store === storeAtModal)
assert(store === storeAtActions)
```
